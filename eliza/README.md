# ELIZA

*Joseph Weizenbaum, 1966 · symbolic AI, pattern matching*

Every other technique in this repository counts something — words, transitions,
frequencies. ELIZA counts nothing. It is a **rule-based** system: a fixed list of
patterns, each paired with a canned response, applied to whatever you type. There
is no corpus, no training, no statistics anywhere in it — and that is precisely
why it belongs first in this story, not last. ELIZA is the losing side of an
argument this whole repository is secretly about: can language be handled by
*writing down the rules*, or does it require *counting from data*? The rest of
this repository is the statistical answer. ELIZA is the rule-based one, and
watching it break is the fastest way to understand why the field moved on.

## The idea

Weizenbaum built ELIZA to play a Rogerian psychotherapist — a role chosen
deliberately, because a Rogerian therapist mostly reflects the patient's own
words back as questions, which needs no world knowledge at all. Say "I need
help" and the therapist doesn't need to know what help means; it just needs to
ask "why do you need help?"

Mechanically, ELIZA is a list of **rules**, checked top to bottom. Each rule
pairs a regular expression against a set of response templates:

```
pattern:    /\bi need (.*)/i
responses:  [ "Why do you need {1}?",
              "Would it really help you to get {1}?",
              "Are you sure you need {1}?" ]
```

The first rule whose pattern matches your input wins. Whatever text the pattern
*captured* — the `(.*)` — gets **pronoun-reflected** ("I" → "you", "my" → "your",
"am" → "are") and spliced into the `{1}` slot of the chosen response:

```
you   > I need some help with my life.
eliza > Why do you need some help with your life?
```

"I" became "you", "my" became "your" — the sentence was never understood, only
grammatically mirrored. When no rule matches at all, ELIZA falls back to a
cycling list of contentless prompts ("Please go on.", "Tell me more.") that keep
the conversation moving without engaging with a single word you said.

### A short aside: rules vs. statistics

For much of the 20th century this was a live argument in linguistics, sharpened
by Noam Chomsky's 1957 critique of purely statistical models of grammar:
language, the argument went, is generated by a finite set of formal rules, so
the right way to build a machine that handles language is to *write down the
rules*, the way ELIZA does. The rival view — the one this whole repository
traces — is that language is too irregular, too context-dependent, and too vast
to enumerate by hand, so the machine should instead *learn from examples*. ELIZA
is a working artifact of the rules camp: remarkably effective for its narrow
trick, and a clear demonstration of where that approach runs out of road.

## What the program builds

The "model" is just the ordered rule list — 17 patterns plus a fallback pool,
all defined openly in [`core.js`](./core.js). There is no data structure to
learn, because there is no learning:

```js
{ pattern: /\bi feel (.*)/i, responses: [
  'Tell me more about that feeling.',
  'Do you often feel {1}?',
  'What do you think is behind feeling {1}?',
] }
```

Response selection **cycles** through each rule's own list rather than drawing
randomly, so a transcript reproduces exactly on every run — appropriate for a
system that has nothing to be random about.

## Worked example (real output)

Running the canned demo conversation:

```
you    > I need some help with my life.
eliza  > Why do you need some help with your life?   [rule #1]
you    > I can't sleep lately.
eliza  > How do you know you can't sleep lately?   [rule #3]
you    > I feel like nobody understands me.
eliza  > Tell me more about that feeling.   [rule #4]
you    > My mother never listens to me.
eliza  > Tell me more about your family.   [rule #11]
you    > Why don't you just tell me what to do?
eliza  > Do you really think I don't just tell you what to do?   [rule #6]
you    > Because I never learned how.
eliza  > Is that the real reason?   [rule #10]
you    > Are you even a real doctor?
eliza  > Why does it matter whether I am even a real doctor?   [rule #8]
you    > Sorry, that was rude.
eliza  > Please don't apologize.   [rule #12]
you    > Yes, I think I should apologize more.
eliza  > You seem quite positive.   [rule #16]
```

Read a few of these closely and the trick both works and falls apart at once.
"I can't sleep lately" reflects cleanly into a probing question. But "Because I
never learned how" gets a generic "Is that the real reason?" — ELIZA has no idea
*what* reason, because it never parsed one. It only recognised the word
"because" and fired a template that happens to sound plausible after any
justification.

## Where it falls short

- **It understands nothing.** Every reply is a syntactic reflection of your own
  words. ELIZA has no model of what "help", "sleep", or "family" *mean* — it
  cannot answer a factual question, hold a fact across turns, or notice a
  contradiction.
- **Coverage does not scale.** Weizenbaum's original DOCTOR script had a few
  hundred patterns; this one has 17. Getting from 17 to "handles arbitrary
  English" is not a matter of writing more rules — natural language has combinatorially
  many ways to say the same thing, and hand-authored patterns cannot keep up. You
  would need infinitely many rules to cover finitely many ideas expressed
  infinitely many ways.
- **It is brittle to phrasing.** "I need help" triggers a rule; "help is
  something I need" does not, even though they mean the same thing. A
  rule-based system only recognises the exact surface forms someone thought to
  write down.
- **The illusion depends entirely on the topic.** Rogerian therapy is a
  best-case scenario for pattern reflection — the therapist is *supposed* to
  say very little. Ask ELIZA to do anything that requires actual knowledge
  (translate a sentence, answer a trivia question, summarise a paragraph) and it
  has nothing to fall back on but the same handful of empty prompts.

That last point is the whole argument for what follows: you cannot hand-write
enough rules to cover language, so **count instead**. A Markov chain never
"understands" a word either, but it needs zero rules — just a text to read and
tally. See [`../markov/`](../markov/) for what counting buys you that ELIZA's
hand-written patterns cannot.

## Usage

```bash
# Run the canned demo conversation (deterministic, no arguments needed)
node eliza/index.js

# Talk to it directly with one line of your own
node eliza/index.js "I need some help with my life"
node eliza/index.js "what should I do"
```

Each line of output shows ELIZA's reply and, in brackets, which rule fired (or
`fallback` if none did) — the whole mechanism is visible, not hidden behind a
black box.
